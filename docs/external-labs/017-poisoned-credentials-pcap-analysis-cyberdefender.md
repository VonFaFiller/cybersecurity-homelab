# Poisoned Credentials Lab – PCAP Analysis (CyberDefenders)

## Objective
Analyze captured network traffic to identify LLMNR/NBT-NS poisoning activity, determine the rogue machine, identify affected hosts, and trace the SMB/NTLM authentication flow that exposed user credentials.

## Scenario
The exercise is based on a PCAP containing NBNS, SMB2, and NTLM-related traffic generated during a poisoning attack in a Windows network.
The goal is to reconstruct how the attacker abused local name resolution protocols to trigger SMB authentication and capture credentials from victim systems.

## References
- https://cyberdefenders.org/blueteam-ctf-challenges/achievements/VonFaFiller/poisonedcredentials/
- https://cyberdefenders.org/walkthroughs/poisonedcredentials/

## Tools Used
- Wireshark
- NBNS filters
- SMB2 packet inspection
- NTLMSSP field analysis

## Investigation Workflow
1. Isolated the initial mistyped NBNS query generated by the first victim.
2. Traced the malicious NBNS response to identify the rogue machine.
3. Checked which other hosts received poisoned responses from the same rogue IP.
4. Pivoted from NBNS traffic to SMB2 traffic directed to the rogue machine.
5. Reviewed the NTLM authentication exchange to identify the compromised username.
6. Examined the NTLMSSP challenge fields to identify the hostname of the machine later accessed via SMB.

## Key Evidence
- Mistyped query source: `192.168.232.162`
- Mistyped NBNS name: `FILESHAARE<20>`
- Rogue machine IP: `192.168.232.215`
- Second affected machine: `192.168.232.176`
- Compromised username: `janesmith`
- NTLM / SMB target host: `ACCOUNTINGPC`
- DNS computer name observed in target info: `AccountingPC.cybercactus.local`

## Findings
- The poisoning activity began with a mistyped NBNS query sent by:
  - `192.168.232.162`
- The mistyped queried name was:
  - `FILESHAARE<20>`
- The rogue machine responding to poisoned name resolution traffic was:
  - `192.168.232.215`
- A second machine that received poisoned responses from the same rogue system was:
  - `192.168.232.176`
- SMB2 traffic directed to the rogue host showed NTLM authentication activity.
- Within the NTLM authentication exchange, the compromised username was:
  - `janesmith`
- The host later identified in the NTLMSSP challenge target information was:
  - `ACCOUNTINGPC`
- The fully qualified DNS computer name shown in the packet details was:
  - `AccountingPC.cybercactus.local`

## Result
- Victim Querying Host: `192.168.232.162`
- Mistyped Query: `FILESHAARE<20>`
- Rogue Machine IP: `192.168.232.215`
- Second Affected Machine: `192.168.232.176`
- Compromised Username: `janesmith`
- Accessed SMB Hostname: `ACCOUNTINGPC`
- DNS Computer Name: `AccountingPC.cybercactus.local`
- Attack Vector: `LLMNR/NBT-NS poisoning leading to SMB credential interception`

## Mistakes / Friction Points
- The main difficulty was not always understanding where to start from the question itself.
- I could usually continue once I had the right protocol or packet type, but I got stuck when I had to choose the first pivot on my own.
- The biggest confusion was in the last question:
  - I initially interpreted it as the hostname of the attacker
  - but the question was actually asking for the hostname of the machine the attacker later accessed via SMB
- I also mixed up different SMB stages:
  - `Session Setup` helped identify the username
  - but it did not answer the hostname question directly
- I initially tried to extract the answer from Tree Connect style evidence, but packet gaps and missing segments made that path unreliable in this PCAP.
- The correct pivot for the last question was the NTLMSSP challenge target information, especially:
  - `ntlmssp.challenge.target_info`
  - `NetBIOS Computer Name`
  - `DNS Computer Name`
