# Poisoned Credentials Lab – PCAP Analysis (CyberDefenders)

## Objective

Analyze captured network traffic to identify LLMNR/NBT-NS poisoning activity, determine the rogue machine, identify affected hosts, and trace the SMB authentication activity that exposed user credentials.

## Scenario

The exercise is based on a PCAP containing NBNS, SMB2, and NTLM-related traffic generated during a poisoning attack in a Windows network.  
The objective is to reconstruct how the attacker abused local name resolution protocols to capture credentials and identify the systems involved in the attack chain.

## References

- https://cyberdefenders.org/blueteam-ctf-challenges/achievements/VonFaFiller/poisonedcredentials/
- https://cyberdefenders.org/walkthroughs/poisonedcredentials/

## Observations

- The analysis was performed in Wireshark on the provided PCAP.
- The attack involved LLMNR/NBT-NS poisoning behavior.
- A mistyped NBNS query was generated by:
  - `192.168.232.162`
- The mistyped queried name was:
  - `FILESHAARE<20>`
- A rogue machine responded to the poisoned name resolution requests:
  - `192.168.232.215`
- A second machine that received poisoned responses from the rogue machine was identified:
  - `192.168.232.176`
- SMB2 traffic directed to the rogue machine showed NTLM authentication activity.
- Within the SMB2 / NTLMSSP authentication exchange, the compromised username was identified:
  - `janesmith`
- The NTLM challenge / target information indicated the hostname of the machine accessed via SMB:
  - `AccountingPC.cybercactus.local`
- The attack chain showed the typical poisoning sequence:
  - mistyped name query
  - malicious NBNS response
  - SMB authentication toward rogue system
  - credential exposure

## Result

- Victim Querying Host: `192.168.232.162`
- Mistyped Query: `FILESHAARE<20>`
- Rogue Machine IP: `192.168.232.215`
- Second Affected Machine: `192.168.232.176`
- Compromised Username: `janesmith`
- Accessed SMB Hostname: `AccountingPC.cybercactus.local`
- Attack Vector: `LLMNR/NBT-NS poisoning leading to SMB credential interception`

## Notes

- The exercise was useful for distinguishing between different protocol roles:
  - NBNS query generation
  - poisoned NBNS response
  - SMB session establishment
  - NTLM credential exposure
- The main analytical difficulty was distinguishing:
  - legitimate broadcast name resolution traffic
  - malicious spoofed NBNS responses
  - SMB traffic used only for authentication follow-up
- Another important distinction was between:
  - the rogue machine IP
  - the victim host sending credentials
  - the SMB target hostname shown in NTLM target information
- A useful improvement for future investigations is to track each stage of the chain in order:
  - query
  - spoofed response
  - SMB setup
  - NTLM fields extracted from packet details
